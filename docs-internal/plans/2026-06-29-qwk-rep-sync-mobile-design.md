# QWK/REP Synchronet Compatibility and Mobile Packet Transport

**Date:** 2026-06-29
**Status:** Design
**Scope:** Bring ViSiON/3 QWK/REP support much closer to the Synchronet behavior and define a transport strategy for a non-web mobile offline-reader workflow.

---

## Problem

ViSiON/3 already has working QWK download/upload plumbing:

- packet writer in `internal/qwk/writer.go`
- REP reader in `internal/qwk/reader.go`
- menu integration in `internal/menu/qwk_handler.go`
- sysop/user docs in `docs/sysop/messages/qwk.md`

That implementation is intentionally minimal. It is enough for simple offline reading and plain reply posting, but it does not yet preserve much of the metadata and compatibility behavior that real-world QWK readers expect from Synchronet-class systems.

The larger product goal is mobile offline message access without building a browser UI. Users should be able to:

1. authenticate from a mobile app
2. request/download a QWK packet
3. read and compose replies offline
4. upload a REP packet back to ViSiON/3

The QWK/REP core must be solid before a mobile transport is worth building.

---

## External Reference Baseline

Primary target behavior is the Synchronet QWK reference:

- Synchronet `ref:qwk` wiki page, last modified **2025-05-04**
- Synchronet source tree under `src/sbbs3/` for implementation checks where the wiki is ambiguous
- MultiMail reader source tree as a practical client-side interoperability reference

Important compatibility items from the Synchronet reference:

- `CONTROL.DAT` layout
- `MESSAGES.DAT` block format
- standard `*.NDX` indexes
- optional `PERSONAL.NDX`
- optional QWKE extensions via `HEADERS.DAT`
- Synchronet-specific extensions via `@MSGID`, `@REPLY`, `@TZ`, `@VIA`, `@REPLYTO`, `@REPLYADDR`
- reply packet first block containing the destination BBS ID
- REP payload file named `ID.MSG`
- support for a logical email/private-mail conference in addition to public conferences

The goal is not byte-for-byte cloning of Synchronet internals. The goal is interoperability with mainstream QWK readers and writers that already work well with Synchronet.

MultiMail is especially useful here because it represents a real offline reader implementation, not just a BBS-side packet producer/consumer. ViSiON/3 should be tested against both:

- Synchronet as a BBS/spec reference
- MultiMail as a reader/reply-workflow reference

## Current State

### What ViSiON/3 already does well

- Builds ZIP-based `.QWK` packets with `CONTROL.DAT`, `DOOR.ID`, `MESSAGES.DAT`, per-conference `.NDX`, and `PERSONAL.NDX`
- Packs unread messages from tagged areas
- Advances last-read pointers only after a successful transfer
- Accepts `.REP` uploads and posts replies into message areas
- Reuses the existing file transfer subsystem instead of inventing a new terminal transfer path

### Current implementation limits

These are the high-value gaps visible in the current code:

1. `internal/qwk/writer.go` does not emit `HEADERS.DAT`.
2. `internal/qwk/reader.go` ignores `HEADERS.DAT` entirely.
3. REP parsing only extracts `Conference`, `To`, `Subject`, and `Body`.
4. REP import in `internal/menu/qwk_handler.go` discards reply/thread metadata and always posts with `AddMessage(...)`.
5. REP import does not preserve private-mail semantics.
6. The first REP block is treated as a spacer and is not validated against the destination BBS ID.
7. Conference mapping currently uses local `MessageArea.ID` directly.
8. The 5-byte NDX record stores only the low 8 bits of the conference number.
9. The packet writer exposes only `DOOR.ID`, not `TOREADER.EXT` / `TODOOR.EXT` style extension files.
10. The QWK BBS ID derivation in `internal/menu/qwk_handler.go` does not enforce Synchronet-style alphabetic-leading ID rules.
11. Private mail is not modeled as a first-class QWK/REP routing case.
12. The current tests are unit-only; there are no fixture-based compatibility tests against packets generated by external readers/BBSes.

This means the current implementation is "basic QWK works" rather than "Synchronet-compatible QWK is trustworthy."

## Goals

### Primary goals

- Match Synchronet packet structure and semantics closely enough that existing QWK readers behave predictably.
- Preserve message intent on upload:
  - public reply vs private reply
  - reply/thread linkage
  - netmail-style destination metadata where applicable
  - authored timestamp/timezone when available
- Keep QWK packet generation/import as an internal service, not menu-only logic.
- Make the same service usable from terminal flows and a future mobile transport.

### Non-goals

- Building a browser-based web reader
- Replacing the SSH/telnet UI
- Replacing JAM as the local message store
- Implementing every obscure Synchronet extension in phase 1

---

## Design Principles

- **Compatibility first:** prefer Synchronet-compatible behavior over a custom ViSiON/3-only interpretation.
- **Service extraction:** packet assembly/import should live in `internal/qwk` plus a thin orchestration layer, not in menu handlers.
- **Stable conference mapping:** never depend on incidental local IDs if a stable exported mapping is needed.
- **Idempotent imports:** REP processing must tolerate reconnects and duplicate uploads.
- **Transport independence:** the mobile transport should only move packets and authenticate users; it should not contain message business logic.

## Proposed Architecture

## 1. Expand `internal/qwk` from codec to full packet model

Add explicit structures for:

- packet manifest
- conference map
- exported-message metadata
- extended header fields
- reply/import result

Suggested new types:

```go
type ExportPacket struct {
    BBSID       string
    UserHandle  string
    Conferences []ConferenceInfo
    Messages    []PacketMessage
    Headers     []ExtendedHeader
}

type PacketMessage struct {
    Conference   int
    Number       int
    From         string
    To           string
    Subject      string
    DateTime     time.Time
    Body         string
    Private      bool
    ReplyMsgID   string
    MessageID    string
    ToNetAddr    string
    FromNetAddr  string
}

type REPMessage struct {
    Conference   int
    To           string
    Subject      string
    Body         string
    Private      bool
    ReplyMsgID   string
    MessageID    string
    ToNetAddr    string
    FromNetAddr  string
    TimeZone     string
}
```

The exact type layout can differ, but the service needs those concepts.

## 2. Add QWKE `HEADERS.DAT` support

Phase-1 extended-header support should include the fields most important to interoperability:

- `Conference`
- `Message-ID`
- `In-Reply-To`
- `When-Written`
- `When-Imported`
- `Sender`
- `Recipient`
- `Subject`
- `Status`
- `To/From network address` where present

Also parse and emit Synchronet's legacy tag-style extensions when encountered:

- `@MSGID`
- `@REPLY`
- `@TZ`
- `@VIA`
- `@REPLYTO`
- `@REPLYADDR`

Rule:

- when both legacy tag lines and `HEADERS.DAT` are present, prefer `HEADERS.DAT`
- preserve legacy support because some readers still emit it

## 3. Introduce a stable QWK conference map

Do not rely on local `MessageArea.ID` as the long-term exported conference identity.

Add a persisted QWK conference map in config or generated metadata:

```go
type QWKConferenceMapEntry struct {
    QWKNumber int    `json:"qwk_number"`
    AreaTag    string `json:"area_tag"`
    Kind       string `json:"kind"` // public, private_mail, netmail
}
```

Behavior:

- reserve `0` for private mail / email compatibility
- assign stable positive conference numbers for public areas
- store by `AreaTag`, not by mutable display name or local numeric ID
- never renumber existing exported conferences automatically once assigned

Why:

- area IDs are an internal implementation detail
- 5-byte `.NDX` records only retain one conference byte in many reader expectations
- stable mapping matters for offline readers, saved reply packets, and future API clients

## 4. Treat private mail as a first-class QWK case

Private mail should not be "just another area if the user happened to tag it."

Export behavior:

- include private mail in conference `0` / email semantics
- include messages addressed to or from the current user only
- populate `PERSONAL.NDX`

Import behavior:

- replies targeting private-mail/email conference route to `AddPrivateMessage(...)`
- preserve private status from REP metadata

This is an important correctness fix, not just a compatibility nice-to-have.

## 5. Extract a QWK service layer above the codec

Create an orchestration package, likely `internal/qwkservice` or an expanded `internal/qwk` service section, responsible for:

- selecting exportable areas for a user
- mapping areas to QWK conference numbers
- gathering messages and metadata from JAM
- building packets
- validating/importing REP uploads
- routing imported messages to:
  - public message posting
  - private mail posting
  - netmail posting if supported later
- updating last-read pointers after successful export
- deduplicating REP imports

Menu handlers should call the service, not implement packet semantics themselves.

## 6. Add REP import validation and deduplication

REP import should validate:

- archive contains the expected `ID.MSG`
- first block BBS ID matches this system's QWK ID
- packet is structurally valid
- conference exists and is writable by the user
- private/public routing is coherent

Add deduplication keyed by a packet fingerprint, for example:

- uploader handle
- packet BBS ID
- SHA-256 of `ID.MSG`

Persist recently seen REP fingerprints in a transactional store. Because a
mobile client may retry an upload while a prior attempt is still being
processed, the dedup check-and-record must be atomic. Standardize on SQLite (or
another store with a real compare-and-set), or, if a flat-file store is used,
explicitly enforce a single-writer model (e.g. a process-wide mutex plus an
atomic write-rename) so an overlapping upload cannot race past the check. A bare
"JSON file" without one of these guarantees is not sufficient.

This prevents accidental double-posting when a mobile client retries an upload.

## 7. Preserve reply/thread linkage

When exporting, include enough metadata for a reader to reference the original message:

- original message ID
- original reply ID if present
- logical conference
- logical message number

When importing:

- map `In-Reply-To` or `@REPLY` back to local `replyToMsgID`
- if the original local JAM message can be found, also set the JAM reply chain where possible

If exact parent lookup fails, the message should still post, but the import result should record that threading degraded.

## Synchronet Compatibility Tiers

## Tier 1: Required for first serious interoperability pass

- valid `CONTROL.DAT`
- valid `MESSAGES.DAT`
- valid per-conference `.NDX`
- `PERSONAL.NDX`
- REP first-block BBS ID validation
- stable conference mapping
- private mail conference handling
- `HEADERS.DAT` read/write for reply metadata
- legacy Synchronet tag parsing for `@MSGID` / `@REPLY` / `@TZ`
- fixture-based tests using real-world packet samples, including packets verified with MultiMail

## Tier 2: Strongly recommended

- `TOREADER.EXT` / `TODOOR.EXT` style capability signaling as needed
- richer network-address fields
- better timezone normalization
- packet manifests/debug output for sysop support
- import reports that show skipped/posted/degraded-thread replies

## Tier 3: Optional

- file-attach semantics if ever needed
- packet compression tuning
- very old reader quirks that no modern client actually depends on

## Mobile Transport Options

## Option A: Reuse terminal session + file transfer protocols

Flow:

1. mobile app opens SSH or telnet session
2. app logs in
3. app drives menu commands
4. app sends/receives `.QWK` and `.REP` through existing transfer protocol support

Pros:

- no new server surface area
- reuses existing auth and transfer code

Cons:

- poor fit for mobile SDKs
- ZMODEM-style tooling is awkward in React Native
- hard to make reliable in background/mobile network conditions
- unnecessarily couples app logic to terminal automation

Conclusion: not recommended as the primary mobile architecture.

## Option B: Add a narrow authenticated QWK packet API

Flow:

1. mobile app authenticates with username/password plus app token or session token
2. app requests a new packet build
3. server returns packet metadata and a download URL or direct payload
4. app opens the `.QWK` in an embedded QWK engine or external QWK reader module
5. app uploads the `.REP`
6. server returns a structured import report

Pros:

- best fit for React Native
- resilient to flaky mobile connectivity
- does not require terminal emulation
- keeps the client focused on packet transport, not BBS screen scraping

Cons:

- introduces a new authenticated network surface
- must be carefully scoped so it does not turn into a general-purpose web frontend

Conclusion: recommended.

## Option C: Add an SSH subsystem or custom packet socket protocol

Pros:

- avoids HTTP if that is philosophically important
- still cleaner than screen-scraping terminal menus

Cons:

- more custom client/server work than HTTPS
- weaker library support in React Native
- little user benefit over a narrow HTTPS transport

Conclusion: acceptable fallback if HTTP is rejected, but second choice.

## Recommended Mobile Design

Implement a small **QWK Packet Service API**, not a web UI.

Recommended endpoints:

- `POST /api/qwk/login`
- `POST /api/qwk/packets`
- `GET /api/qwk/packets/{id}`
- `GET /api/qwk/packets/{id}/download`
- `POST /api/qwk/packets/{id}/rep`

Or, if avoiding job resources:

- `POST /api/qwk/export`
- `POST /api/qwk/import`

Minimum response data:

```json
{
  "bbs_id": "VISION3",
  "packet_id": "uuid",
  "message_count": 142,
  "conference_count": 18,
  "generated_at": "2026-06-29T19:30:00Z"
}
```

Import result:

```json
{
  "posted": 12,
  "skipped": 2,
  "duplicates": 1,
  "thread_link_failures": 1,
  "results": []
}
```

Constraints:

- disabled by default in config
- packet-only scope, no message browsing endpoints
- token/session auth distinct from sysop web tools
- rate-limited
- full audit logging

This preserves the "not a web frontend" goal while giving a mobile app the transport it actually needs.

## React Native App Shape

The mobile app should not depend on terminal automation.

Preferred client architecture:

1. **Auth module**
   - username/password
   - optional saved app token

2. **Packet sync module**
   - request export
   - download `.QWK`
   - upload `.REP`
   - retry-safe with packet/import IDs

3. **Local QWK engine**
   - either embed a JS/TS QWK parser/writer
   - or wrap an existing reader core if a suitable one exists

4. **Offline store**
   - local SQLite or file-backed packet cache
   - keep packet metadata, message indexes, reply drafts

5. **Reply compiler**
   - build `.REP` exactly as ViSiON/3 expects
   - preserve `HEADERS.DAT` where supported

The app can remain entirely packet-oriented. It does not need online message browsing endpoints.

Expand `REPMessage` beyond the current fields:

- `Private`
- `ReplyMsgID`
- `MessageID`
- `ToNetAddr`
- `FromNetAddr`

## Story 4: Private-mail aware import routing

**Goal:** do not post uploaded private replies through the public posting path.

Import behavior should branch:

- private-mail/email conference -> `AddPrivateMessage(...)`
- public conference -> `AddMessage(...)` or `AddMessageWithDate(...)`

This is the first hard correctness fix because the current code always uses:

- `e.MessageMgr.AddMessage(...)` in `internal/menu/qwk_handler.go`

Primary files:

- `internal/menu/qwk_handler.go`
- extracted QWK service
- `internal/message/manager.go` if new helper methods are needed

## Story 5: Reply/thread linkage preservation

**Goal:** preserve reply chains when readers emit them.

Import should attempt:

1. `HEADERS.DAT` `In-Reply-To`
2. Synchronet legacy `@REPLY`
3. fallback to no parent if neither resolves

If the parent is found by message ID, pass that through to the message manager so JAM metadata is preserved as far as existing APIs allow.

Primary files:

- `internal/qwk/reader.go`
- `internal/qwk/headers.go`
- `internal/message/manager.go`

## Story 6: REP first-block validation

**Goal:** validate the REP destination and reject packets meant for another BBS.

The current parser skips the first block as a spacer. Synchronet-compatible behavior should instead:

- read the first block
- extract the destination BBS ID
- compare it with the expected local QWK ID

Primary files:

- `internal/qwk/reader.go`
- `internal/qwk/reader_test.go`

## Story 7: `HEADERS.DAT` write support

**Goal:** export enough metadata for MultiMail and similar readers to reply cleanly.

Writer work:

- emit `HEADERS.DAT`
- include message IDs, reply IDs, timestamps, and privacy status
- decide precedence rules when the packet also carries legacy Synchronet tags in body/header content

Primary files:

- `internal/qwk/writer.go`
- new `internal/qwk/headers.go`
- `internal/qwk/writer_test.go`

## Story 8: `HEADERS.DAT` and legacy tag read support

**Goal:** import replies from real readers, not just from ViSiON/3-generated test packets.

Reader work:

- parse `HEADERS.DAT`
- parse Synchronet tags from message text/header regions where needed
- normalize both into the expanded `REPMessage`

Primary files:

- `internal/qwk/reader.go`
- `internal/qwk/reader_test.go`
- fixture tests under `internal/qwk/testdata/`

## Story 9: REP deduplication

**Goal:** make uploads retry-safe for mobile clients.

Suggested implementation:

- hash `ID.MSG`
- key dedup on `user + bbs_id + hash`
- persist recent fingerprints in a store whose check-and-record is atomic
  (SQLite, or a flat file guarded by a single-writer model — see the REP
  validation/dedup notes above); the dedup guarantee must hold under
  overlapping retries, not just sequential uploads

Primary files:

- `internal/qwk/import_dedup.go`
- QWK service

## Story 10: Packet API

**Goal:** expose packet transport without creating a browsing frontend.

Suggested package:

- `internal/qwkapi/`

The API should call the same service used by the menu handlers. No packet semantics should live only in HTTP handlers.

Primary files:

- `cmd/vision3/main.go`
- `internal/qwkapi/handler.go`
- extracted QWK service
- config structs under `internal/config/`

## Open Questions

1. Should ViSiON/3 reserve conference `0` strictly for private mail/email, even if some current packets already expose area ID `0` differently?

2. Should the stable QWK conference map live in `data/` as runtime state or in `configs/` as sysop-managed state?

3. Do we want imported reply timestamps preserved via `AddMessageWithDate(...)` when a reader provides trustworthy authored dates, or should all REP imports use server receive time?

4. Should packet API authentication reuse the existing login/password flow to mint short-lived tokens, or should it have a dedicated app-password concept?

5. Do we want the first mobile client to embed a QWK engine directly, or should it hand `.QWK` / `.REP` files to an external reader app where available?

## Risks

- QWK readers vary in how faithfully they preserve extended metadata.
- Overfitting to one reader can break others; fixture diversity matters.
- Conference numbering changes can strand old offline packets if mapping is not stable.
- A mobile API can drift into a general-purpose message API unless its scope is kept narrow.
